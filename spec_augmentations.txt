2. Assume single user.
3. For attempts, scheduling and mutable state use SQLite. Markdown/YAML should be the source of truth for human authored learning content or reference. SQLite should be an event log + scheduling + computed learner state. If the user or codex should comfortably edit it and it needs to be human readable use markdown or yaml. If its a component that needs consistency, or if its history or scheduling put it in SQLite. Markdown is for narrative memory (goals.md, subject.md, notes.md, errors.md, reflection.md, sessions/*.md, long form notes, error notebooks, codex readable context, learner profile, goals). YAML is good for concept graphs, prerequisite graphs, confusable concepts, domain templates, knowledge type, concept membership, worked example meta data, difficuolty priors, cards. SQLite should be for canonical states like attempts, review schedule, FSRS state, mastery state, generated item registry, error events, session metadata, card important hashes, active/dormant status, due queue. You can also have codex read curated generated context files such as: exports/
  current-state.md
  due-today.yaml
  weak-concepts.yaml
  recent-errors.md
  mastery-snapshot.yaml
Due reviews, suggested session time, weak concepts, recent recurring errors, recommended next actions. Generated cards should go through an inbox to be accepted. Cards should additionally have provenance to discriminate between ai generated content and human authored content. 
4. Use py-fsrs.
5. A Learning Object is the stable thing being learned not the exact prompt shown on a given day. So its a durable unit of knowledge or skill that has a stable identity across multiple practice formats. 1 learning object can have many practice modes. For implementation, I would store each concrete prompt as its own YAML item linked to a parent learning_object_id. For example, 

Concept:
  Fine-Gray competing risks

Learning Object:
  Fine-Gray subdistribution hazard definition

Practice Items:
  - Define it from memory.
  - Explain it in plain English.
  - Contrast it with cause-specific hazard.
  - Apply it to a bail/disposition example.
  - Identify which statement is wrong.

 Each Practice Item should have 1 primary practice_mode.

Each practice item should say which dimensions it measures. For now we can use EMA per dimension as a basic update rule unless you have a better idea. It should also be difficulty aware mastery update where hard success should raise mastery more while easy failure should lower mastery more. We should also include error aware cross dimension effects since some errors reveal weaknesses in dimensions that the item did not explicitly target. For example, 

Prompt type: transfer
Error: forgot definition

should update bough transfer and recall (lower both). 

Another example:
Prompt type: retrieval
Error: correct definition but bad application

Should not heavily penalize recall but penalize schema/transfer.

You can have error impact maps

error_impacts:
  recall_failure:
    recall: -0.25

  conceptual_error:
    schema: -0.25
    transfer: -0.10

  theorem_selection_error:
    schema: -0.30
    transfer: -0.20

  procedure_error:
    procedure: -0.25
    fluency: -0.05

  transfer_failure:
    transfer: -0.30
    schema: -0.10

  high_confidence_wrong:
    schema: -0.20

and apply cross dimensional corrections. 

Additionally, make sure to flag high confidence wrong answers since it's not just a failed recall but a misconception. It should lower schema, confidence_calibration, and increase error notebook. 
For fluency, you can compute it from number of pauses/hints, consistency across attempts, latency relative to expected time.

For most free text answers, 0-4 rubric should be sufficient.

0 = no meaningful answer / "I don't know"
1 = mostly wrong, but contains a relevant fragment
2 = partially correct, missing key idea or has serious confusion
3 = mostly correct, minor omission/imprecision
4 = fully correct for this prompt

def score_to_fsrs_rating(score: float) -> str:
    if score < 0.30:
        return "again"
    if score < 0.65:
        return "hard"
    if score < 0.90:
        return "good"
    return "easy"
